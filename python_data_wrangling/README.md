# Introduction

- **Business context**

  
London Gift Shop (LGS) is a UK-based online retailer specializing in giftware products. Although the company has operated its online store for several years, revenue growth has slowed in recent periods. A significant portion of LGS customers are wholesalers, which makes understanding customer purchasing behavior especially critical for designing effective marketing strategies.


  The LGS marketing team wants to use historical transaction data to gain insight into how customers shop, which products generate the most value, and how purchasing patterns differ across customer segments. These insights will support data-driven sales and marketing decisions.

- **How LGS would use the analytics results**

  The analytics results from this project help the LGS marketing team:

    - identify high-value, loyal, and at-risk customers through RFM segmentation,
    - understand customer purchase frequency, recency, and spending behavior over time,
    - track monthly sales growth and detect seasonal patterns,
    - design targeted marketing campaigns such as personalized promotions for Champions, win-back campaigns for the "Can't Lose" segment, and re-engagement offers for Hibernating customers.

  By using these insights, LGS can improve customer retention, increase average order value, and drive revenue growth.


- **Work description and technologies used**

  In this PoC project, historical transaction data is analyzed using Python-based data analytics tools. The work includes data loading from both a PostgreSQL database and a CSV file, data cleaning, feature engineering, exploratory analysis, and RFM-based customer segmentation.

  The main technologies used are:

    - **Jupyter Notebook** - end-to-end analysis and visualization
    - **Python 3** - core programming language
    - **Pandas and NumPy** - data wrangling, feature engineering, and aggregation
    - **Matplotlib** - data visualization
    - **PostgreSQL + Docker** - upstream data storage (data loaded via SQLAlchemy)
    - **SQLAlchemy + psycopg2** - database connectivity

# Implementation

## Project Architecture

- **Architecture description**


  This project follows a simple PoC architecture. Transaction data generated by the LGS online store is exported by the LGS IT team into a SQL file, with customer personal information removed. The Jarvis team does not access the LGS cloud environment directly. Instead, the data is loaded into a local PostgreSQL instance running in a Docker container and analyzed using Jupyter Notebook.

## Data Analytics and Wrangling

**Jupyter Notebook:** [`retail_data_analytics_wrangling.ipynb`](./retail_data_analytics_wrangling.ipynb)

The notebook is structured around the following analytical questions:

### 1. Invoice Amount Distribution

Computed the total invoice amount per transaction by multiplying `unit_price` by `quantity`. Key findings:

- The full distribution is heavily right-skewed: Mean (359.65 GBP) > Median (196.34 GBP)
- Mode = 0.00 GBP, caused by canceled orders offsetting their originals
- Extreme outliers reach +/-168,469 GBP (symmetric, confirming they are matched cancel pairs)
- Filtering to the 85th percentile (max 594 GBP) reveals that 50% of invoices fall between 62.70 GBP and 330.00 GBP

### 2. Monthly Placed and Canceled Orders

Identified canceled orders by the `C` prefix in `invoice_no`. Key findings:

- Placed orders consistently exceed cancellations every month
- A clear peak in November 2011 (~2,600 placed orders) driven by holiday shopping
- Canceled orders remain stable at roughly 200-500 per month
- December 2011 shows a drop because the dataset ends mid-month

### 3. Monthly Sales and Sales Growth

Calculated monthly revenue as `unit_price * quantity`, then computed month-over-month growth using `pct_change()`. Visualized both the absolute sales trend and the growth rate on a line chart with a zero-reference line.

### 4. Monthly Active Users

Counted distinct `customer_id` values per month (after dropping null IDs) to track engagement trends over the observation period.

### 5. New vs. Existing Users

Identified each customer's first purchase month and tagged every subsequent transaction as `new` or `existing`. Pivoted the result to visualize the monthly new/existing user split as a stacked bar chart.

### 6. RFM Table

Computed Recency, Frequency, and Monetary metrics per customer using `reference_date = 2012-01-01`:

- **Recency:** days since last purchase
- **Frequency:** number of unique invoices
- **Monetary:** total spend (unit_price * quantity)

### 7. RFM Segmentation

Scored each RFM dimension into quintiles using `pd.qcut`, concatenated the Recency and Frequency scores into an `RFM_SCORE`, and mapped customers to 10 named segments using a `seg_map` regex dictionary.

Three key segments highlighted:

| Segment | Count | Avg Recency | Avg Frequency | Strategy |
|---|---|---|---|---|
| **Champions** | 856 | 30 days | 24 purchases | Reward and retain - they drive most revenue |
| **Can't Lose** | 85 | 343 days | 18 purchases | Win back with personalized discounts |
| **Hibernating** | 1,535 | 487 days | 1 purchase | Re-engage with broad promotional offers |

# Improvements

If more time were available, the following improvements could be made:

1. Refactor the notebook into modular Python scripts to improve maintainability and reusability across projects.

2. Add a cohort retention analysis to track how well LGS retains customers month over month.

3. Implement Customer Lifetime Value (CLV) estimation or a churn prediction model to quantify the revenue impact of each segment.

4. Add automated data quality checks (e.g., negative quantity detection, duplicate invoice validation) and schedule regular data refreshes to support ongoing analytics.

5. Migrate the pipeline from single-machine pandas to a distributed framework (e.g., Spark) to support larger datasets as LGS grows.
